SETUP QUEST · 01 / 06v2.0 · 2026
CSCI-Y100 · WEEK 03 · WHY THIS LESSON

Why now, why this way.

Most CS curricula assume the dev environment Just Works by Week 1. It doesn't. Students arrive with a laptop, a vague memory of a homework instruction page, and no help when their `python` command says "command not found" at 11pm on a Tuesday. The cost shows up six weeks later as a withdrawal.

Setup Quest gets the install done in class, with TAs at every table, before the first project deliverable hits in W4. Three rounds — terminal/editor, runtime/git, Obsidian/library — each with a verified outcome. The room doesn't move on until everyone's machine runs the verification command for that round.

§ A Outcomes — what every student leaves with

  • Working terminal + editor. They can open a terminal, navigate one folder up, and edit a file in VS Code without consulting a tutorial.
  • Working runtime + git. Python (or course language) runs on their machine. They've made one commit, pushed it to a GitHub repo of their own.
  • Working second brain. Obsidian is installed with a course vault. The vault has a folder structure (Weeks/, Project/, Bugs/, Reflections/, Sources/) and a first note in Weeks/.
  • One credible source from an IU library database. Title + link + a one-sentence "why I trust this" line, captured in Sources/. Not a Google result.

Why this beats the usual setup tutorial Setup tutorials get bookmarked and forgotten. A 90-minute supervised install gets every student to "verified working" in the same hour. The TA-to-student ratio (1:8 ideal) is what makes it work — without it, the lab becomes a help-desk queue.
SETUP QUEST · 02 / 06v2.0 · 2026
CSCI-Y100 · WEEK 03 · PRE-CLASS · 24 HOURS OUT

Pre-class checklist.

Setup Quest fails when these don't get done. Run the checklist 24 hours before class.

§ C Room + tech

  • Confirm room has power at every seat. Reserve a backup power strip.
  • Confirm Wi-Fi works under load (15+ simultaneous large downloads). If campus Wi-Fi is shaky, flag it now.
  • Test the projector. Mirror the demo terminal in a font that's readable from the back row (16pt+).
  • Print the OS-specific install handouts (3 versions: macOS / Windows / Linux). One per seat.
  • Pre-stage TA machines with the final install state. They should be the demo machines, not students debugging on the fly.

§ D People

  • Confirm TA-to-student ratio. 1:8 ideal, 1:12 hard floor. Below that, install round 1 leaks past 50 minutes.
  • Coordinate with Wells Library liaison librarian: drop-in (15 min) during § 03 OR a 5-min recorded walkthrough of the database list playing in the background.
  • Share the run-of-show with TAs. They should know which round verifies which outcome and which trouble-shoots are theirs.

§ E Materials

  • OS-specific install handouts (printed)
  • Slides + speaker notes (`setup-quest-slides.html`)
  • TILT rubric (`setup-quest-tilt-rubric.html`)
  • Verification commands cheat sheet (per round)
  • Obsidian vault starter folder structure (zipped — back-up if Internet fails)
  • Bookmark list: IU Libraries home, database list, librarian chat
SETUP QUEST · 03 / 06v2.0 · 2026
CSCI-Y100 · WEEK 03 · RUN OF SHOW · §§ 00–02

Run of show. First half.

§ 00 Why a second brain 8 MIN · COLD OPEN · COMPUTER OFF
+0OPEN — laptops closed, full attention forward
Three weeks ago we found the toolbox. Two weeks ago we found the time. This week we make the third move that 80% of CS students never make on purpose — we externalize. By end of class your laptop is going to know how to run code, and your second brain is going to know how to keep what you learned. Both matter. Skipping the second one is how juniors look up at Week 12 and realize they have no idea what they did in Week 4.

Do

  • Slides 1–3: anchor question + Big Ideas (#2 verbs · #4 externalize).
  • One question to the room: "What did you learn in W2 that you couldn't recall right now without looking?" Wait. Don't fill the silence.
  • Hand to TAs: "We're going to spend the next 80 minutes making sure none of you are that student in W12."
§ 01 Install round 1 — terminal + editor 20 MIN · INSTALL · TAs HANDS-ON
+8HANDOUT — pass the OS-specific install sheets

Facilitator move

  • Project the demo terminal (16pt+ font). Run `echo "hi"` and `pwd`. That's the verification target.
  • Walk through VS Code install — show how to open the terminal pane inside it.
  • Set the timer. 18 minutes for installs, 2 minutes for the verification check.

TA move

  • Stand. Don't sit at one student's desk for more than 2 minutes.
  • If a student is stuck on the same step for > 5 min, escalate — call the facilitator over.
  • Verification: each student runs `echo "hi"` aloud at their own terminal before § 02 starts.

Watch for

PATH issues on Windows. The student installs Python or VS Code, the install "succeeds," but the terminal still says command not found because PATH didn't update. Fix: close + reopen the terminal. If still broken, restart the machine. If still broken, the install didn't add to PATH — re-run installer with "Add to PATH" checked.
§ 02 Install round 2 — runtime + git 15 MIN · INSTALL · COMMIT IS THE GOAL
+28RUN — first script, first commit
A repo with one commit is a repo. A repo with zero commits is a folder. We're making a repo today.

Do

  • Demo: `python --version` (or course-language equivalent). Then a one-line `print("hi")` script. Run it. The room sees output.
  • Demo: `git init`, create a one-line README, `git add`, `git commit -m "first"`. That's the verification target for this round.
  • GitHub: students log in (or sign up — flag this on the install handout if needed). Push the local repo to a new GitHub repo. Repo can be empty besides the README.

Watch for

Git authentication. GitHub no longer accepts password auth for HTTPS — students need a personal access token or SSH key. Have the SSH-key path on the handout (it's faster). Plan for 5–10 minutes of GitHub-account-creation time for any student who doesn't already have one.
SETUP QUEST · 04 / 06v2.0 · 2026
CSCI-Y100 · WEEK 03 · RUN OF SHOW · §§ 03–06

Run of show. Second half.

§ 03 Install round 3 — Obsidian + IU library 15 MIN · INSTALL · LIBRARIAN OPTIONAL
+43VAULT — Obsidian, folder structure, library bookmarks

Do

  • Demo: install Obsidian. Create a vault named `csci-y100`. Set up the folder structure on screen: Weeks/, Project/, Bugs/, Reflections/, Sources/.
  • Demo: bookmark IU Libraries home, the database list, and the chat-with-a-librarian button.
  • Librarian (if scheduled): hand the room over for a 10-min walk-through of the database list. They know it better than you. Don't fight for the mic.
  • Verification: every student opens Obsidian and shows their TA the five folders. TAs check off Start Log entries.

Watch for

The "I'll do it later" face. Obsidian feels less urgent than Python. Counter the bypass directly: "If you skip this, every other lesson's capture template has nowhere to land. Five minutes now or fifteen on your own at 11pm."
§ 04 First note · first source 15 MIN · USE WHAT YOU INSTALLED
+58WRITE — first vault note + first credible source
Installing tools is not a capability. Using them is. We're doing both today, in the same room, while help is at your elbow.

Do

  • First note (Weeks/W03-setup-quest.md): one sentence on what you installed. Save.
  • First source (Sources/...): pull one credible online source — from an IU database, NOT Google — about something you're curious about. Capture title, link, one-sentence "why I trust this." Save.
  • Constraint: the source has to be findable in an IU database. Google Scholar is NOT acceptable here — that's the point. Flagging this is a TA job; bypassing it is a TA conversation.

Watch for

The student who treats this as homework and tries to leave with "I'll find a source later." The whole lesson collapses if they skip § 04. Stand at the door if you have to.
§ 05 Wrap-up + 1:1 troubleshoot 12 MIN · NO STUDENT LEAVES BROKEN
+73UNBLOCK — the most stuck student gets the facilitator

Do

  • TAs sweep the room. Anyone whose Start Log is missing a deliverable gets help.
  • You (facilitator) visit the most-stuck student. Sit. Solve. The optics matter — students remember who got abandoned.
  • If a student's stack is genuinely broken (corrupt install, broken laptop), schedule a 1:1 office hours slot before W4. Don't pretend it's fixed.
§ 06 Take-home 5 MIN · LOG THE ARTIFACTS
+85LOG — three lines, no story
  • Confirm in your Start Log: GitHub repo URL · Obsidian vault path · first source citation.
  • Three lines. No prose. Facilitator spot-checks during 1:1s.
SETUP QUEST · 05 / 06v2.0 · 2026
CSCI-Y100 · WEEK 03 · TROUBLESHOOTING + STUDENT HANDOUT

When things break.

SymptomMost likely fix
"command not found" after install PATH not updated. Close + reopen terminal. If still broken, restart machine. If still broken, re-run installer with "Add to PATH" option (Windows-specific).
Git push asks for password and rejects it GitHub HTTPS auth requires a personal access token, not a password. Switch to SSH (faster) or generate a PAT in GitHub Settings → Developer settings.
Python script runs but says "module not found" Wrong Python install or missing pip package. Check `which python` (or `where python` on Windows). If it points to an unexpected install, alias it or use full path.
Obsidian vault folder doesn't appear in the sidebar Open vault → "Open another vault" → point at the folder. Don't drag-and-drop into the app — that confuses Obsidian's vault detection.
Student can't access an IU database from off-campus VPN required for some databases. Bookmark the IU VPN install page; this is a known recurring issue. Alternative: the library proxy URL prefix (`https://proxy.ulib.iupui.edu/login?url=...`).
Student's laptop is genuinely broken or borrowed-mid-install Loaner program: IU Library has laptops to lend. This is the right call — don't try to fix a hardware problem in a 15-min install round.
