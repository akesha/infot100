① PURPOSE · WHY THIS

Why I'm asking you to do this

In CS, you'll spend more time debugging than writing new code. The same skill works on study habits — once you can name a bug, isolate where it shows up, and test a small fix, you have a tool that outlasts this class.

By the end of this assignment you will be able to:

  • Diagnose a study-habit problem with the same precision you'd use on broken code.
  • Design the smallest possible fix and predict what would prove it works.
  • Test that fix over 7 days using daily evidence, not vibes.
  • Iterate — keep, tweak, or scrap the fix based on what the log actually says.
CONNECTION TO YOUR MAJOR
This is the same loop you'll use in CS 200, in your first internship, and on every group project: observe → hypothesize → test → revise. We're just running it on you instead of code.
② TASK · WHAT TO DO

What to do, exactly

Submit one document (Google Doc, Markdown, PDF — doesn't matter) with the following four parts. Templates are linked in Canvas.

  1. Finalize your bug report
    Use the form from class. Five fields: Title, Symptom, Repro Steps (when does it happen?), Hypothesis (what's actually causing it?), Patch (the smallest fix you'll try). Patch must be doable in ≤ 5 minutes per day.
  2. Run the patch for 7 days
    Each evening, log one line: did the fix happen today? Y / N / N/A + a one-sentence note (why not, what helped, what tripped you up). 90 seconds, no more. Skip-days are fine — log them as N/A with the reason.
  3. Write a 3-sentence reflection
    On day 7, answer in 3 sentences total: (a) What did the fix change? (b) Where did it fail? (c) Will you keep it, tweak it, or scrap it — and why?
  4. Submit on Canvas before next class
    One file. Bring a printed or screenshot copy to class — we'll re-open or close the issue together in groups of three.
⚠ COMMON TRAPS (READ THIS)
Trap 1: Patch too big. "Study 2 hrs every night" is not a fix, it's a wish. A real patch is "open my notes app at 7 pm for 5 min."
Trap 2: Lying to the log. An honest N is worth more than a fake Y. The point is data, not performance.
Trap 3: Calling failure a moral problem. If the patch flopped, the patch was wrong — not you. Reopen the issue with a smaller patch.
